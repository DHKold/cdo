# Requirements Document

## Introduction

This document specifies requirements for three CDo enhancements: (1) a comprehensive README update with feature documentation and a quick-start guide, (2) a virtual environment (`--venv`) feature for `cdo init` that provides full project isolation similar to Python's venv, and (3) a file-locking mechanism for `cdo build` and `cdo test` commands to prevent concurrent build conflicts.

## Glossary

- **CDo**: The C/C++ build system and project manager application written in C.
- **README_Generator**: The component responsible for producing the project README.md content.
- **Venv_Initializer**: The component of `cdo init --venv` that creates the `.cdo` virtual environment directory structure and activation scripts.
- **Activation_Script**: A shell script generated by the Venv_Initializer that modifies the current shell session to use the local CDo environment.
- **Deactivation_Function**: A shell function or script that reverses the environment modifications applied by the Activation_Script.
- **Build_Lock_Manager**: The component that acquires and releases exclusive file locks during build and test operations.
- **Lock_File**: A file created in the workspace `.cdo` directory to represent an exclusive lock on build operations.
- **PAL**: The Platform Abstraction Layer providing cross-platform filesystem, process, path, sysinfo, and time primitives.
- **Lock_Timeout**: The maximum duration in seconds that the Build_Lock_Manager waits for an existing lock to be released before failing.
- **CDO_HOME**: An environment variable set by the Activation_Script pointing to the `.cdo` directory.
- **CDO_VENV**: An environment variable set by the Activation_Script indicating that a virtual environment is active (value: path to the `.cdo` directory).

## Requirements

### Requirement 1: README Feature Documentation

**User Story:** As a CDo user, I want the README to contain complete feature documentation, so that I can understand all available commands and capabilities without reading source code.

#### Acceptance Criteria

1. THE README_Generator SHALL produce a document that includes a description of each CDo command (build, run, test, clean, new, init, deps, catalog, tool, doctor, shader).
2. THE README_Generator SHALL produce a document that includes a description of workspace configuration via `cdo.toml` and per-crate configuration via `crate.toml`.
3. THE README_Generator SHALL produce a document that includes a description of build profiles (debug, release, relwithdebinfo) and their settings.
4. THE README_Generator SHALL produce a document that includes a description of the `--venv` flag for `cdo init`.
5. THE README_Generator SHALL produce a document that includes a description of the build-locking mechanism.

### Requirement 2: README Quick-Start Guide

**User Story:** As a new CDo user, I want a quick-start guide in the README, so that I can go from project creation to a successful build in minimal steps.

#### Acceptance Criteria

1. THE README_Generator SHALL include a quick-start section that demonstrates the workflow from `cdo init` to `cdo build` to `cdo run`.
2. THE README_Generator SHALL include example shell commands showing project initialization with a template.
3. THE README_Generator SHALL include example shell commands showing how to build and run a project.
4. THE README_Generator SHALL include a section describing installation and prerequisites for CDo.

### Requirement 3: Virtual Environment Directory Creation

**User Story:** As a developer, I want `cdo init --venv` to create a self-contained `.cdo` directory with a local CDo binary and activation scripts, so that my project uses an isolated tool environment.

#### Acceptance Criteria

1. WHEN the user invokes `cdo init` with the `--venv` flag, THE Venv_Initializer SHALL create a `.cdo` directory in the current workspace root.
2. WHEN the user invokes `cdo init` with the `--venv` flag, THE Venv_Initializer SHALL copy the currently-running `cdo` executable into the `.cdo` directory.
3. WHEN the user invokes `cdo init` with the `--venv` flag, THE Venv_Initializer SHALL generate activation scripts for Windows batch (`activate.bat`), Windows PowerShell (`activate.ps1`), and Unix shells (`activate.sh`).
4. WHEN the user invokes `cdo init` with the `--venv` flag, THE Venv_Initializer SHALL generate a deactivation mechanism for each supported shell platform.
5. IF the `.cdo` directory already exists when `--venv` is specified, THEN THE Venv_Initializer SHALL preserve existing content (tools, cache) and only regenerate the activation scripts and local CDo binary.

### Requirement 4: Activation Script — Environment Modification

**User Story:** As a developer, I want the activation script to modify my shell environment so that the local CDo binary and tools are used instead of any globally installed version.

#### Acceptance Criteria

1. WHEN the Activation_Script is sourced, THE Activation_Script SHALL prepend the `.cdo` directory to the PATH environment variable.
2. WHEN the Activation_Script is sourced, THE Activation_Script SHALL set the CDO_HOME environment variable to the absolute path of the `.cdo` directory.
3. WHEN the Activation_Script is sourced, THE Activation_Script SHALL set the CDO_VENV environment variable to the absolute path of the `.cdo` directory.
4. WHEN the Activation_Script is sourced, THE Activation_Script SHALL modify the shell prompt to include a `(cdo)` prefix indicating the active virtual environment.
5. WHEN the Activation_Script is sourced, THE Activation_Script SHALL store the original PATH value so that deactivation can restore the previous state.

### Requirement 5: Deactivation — Environment Restoration

**User Story:** As a developer, I want to deactivate the virtual environment cleanly, so that my shell returns to its original state.

#### Acceptance Criteria

1. WHEN the Deactivation_Function is invoked, THE Deactivation_Function SHALL restore the PATH environment variable to the value stored before activation.
2. WHEN the Deactivation_Function is invoked, THE Deactivation_Function SHALL unset the CDO_HOME environment variable.
3. WHEN the Deactivation_Function is invoked, THE Deactivation_Function SHALL unset the CDO_VENV environment variable.
4. WHEN the Deactivation_Function is invoked, THE Deactivation_Function SHALL restore the shell prompt to the value it had before activation.

### Requirement 6: Activation Script — Cross-Platform Support

**User Story:** As a developer working across Windows and Unix platforms, I want platform-appropriate activation scripts, so that the virtual environment works regardless of my operating system or shell.

#### Acceptance Criteria

1. THE Venv_Initializer SHALL generate an `activate.bat` script compatible with Windows CMD.
2. THE Venv_Initializer SHALL generate an `activate.ps1` script compatible with Windows PowerShell 5.1 and later.
3. THE Venv_Initializer SHALL generate an `activate.sh` script compatible with POSIX-compliant shells (bash, sh, zsh).
4. WHEN the Activation_Script is sourced on Windows (CMD), THE Activation_Script SHALL use `set` commands and `%VAR%` syntax for environment manipulation.
5. WHEN the Activation_Script is sourced on Windows (PowerShell), THE Activation_Script SHALL use `$env:VAR` syntax for environment manipulation.
6. WHEN the Activation_Script is sourced on Unix, THE Activation_Script SHALL use `export` commands for environment manipulation.

### Requirement 7: Build Lock Acquisition

**User Story:** As a developer running concurrent terminal sessions, I want `cdo build` and `cdo test` to acquire an exclusive lock before modifying build artifacts, so that parallel invocations do not corrupt the output.

#### Acceptance Criteria

1. WHEN `cdo build` is invoked, THE Build_Lock_Manager SHALL attempt to acquire an exclusive Lock_File in the workspace `.cdo` directory before any compilation begins.
2. WHEN `cdo test` is invoked, THE Build_Lock_Manager SHALL attempt to acquire an exclusive Lock_File in the workspace `.cdo` directory before any compilation begins.
3. WHILE the Lock_File is held by another process, THE Build_Lock_Manager SHALL wait and retry acquisition until the Lock_Timeout is reached.
4. THE Build_Lock_Manager SHALL use a default Lock_Timeout of 30 seconds.
5. IF the Lock_Timeout is reached and the Lock_File is still held, THEN THE Build_Lock_Manager SHALL terminate the command with a non-zero exit code and an error message indicating the lock could not be acquired.

### Requirement 8: Build Lock Release and Cleanup

**User Story:** As a developer, I want the build lock to be released reliably after a build completes or fails, so that subsequent builds are not blocked.

#### Acceptance Criteria

1. WHEN the build or test command completes (success or failure), THE Build_Lock_Manager SHALL release the Lock_File.
2. IF the CDo process terminates abnormally (crash or signal), THEN THE Build_Lock_Manager SHALL ensure the Lock_File is removed or released via operating system file handle semantics.
3. THE Build_Lock_Manager SHALL record the process ID and start timestamp inside the Lock_File content for diagnostic purposes.

### Requirement 9: Build Lock PAL Integration

**User Story:** As a CDo maintainer, I want file locking implemented through the PAL layer, so that the locking mechanism is portable across Windows and Unix.

#### Acceptance Criteria

1. THE PAL SHALL expose a `pal_file_lock_exclusive` function that acquires an exclusive lock on a given file path.
2. THE PAL SHALL expose a `pal_file_lock_release` function that releases a previously acquired lock.
3. WHEN running on Windows, THE PAL SHALL implement file locking using the Windows `LockFileEx` API or equivalent kernel locking mechanism.
4. WHEN running on Unix, THE PAL SHALL implement file locking using `flock()` or `fcntl()` advisory locks.
5. THE PAL SHALL accept a timeout parameter (in milliseconds) for lock acquisition and return `PAL_ERR_TIMEOUT` when the timeout expires without acquiring the lock.

### Requirement 10: Lock Timeout Configuration

**User Story:** As a developer in a CI environment with long builds, I want to configure the lock timeout, so that I can adjust wait behavior for my workflow.

#### Acceptance Criteria

1. WHERE the `--lock-timeout` option is provided to `cdo build` or `cdo test`, THE Build_Lock_Manager SHALL use the specified value (in seconds) as the Lock_Timeout.
2. IF the `--lock-timeout` value is set to 0, THEN THE Build_Lock_Manager SHALL fail immediately when the Lock_File is already held without waiting.
3. IF no `--lock-timeout` option is provided, THEN THE Build_Lock_Manager SHALL use the default Lock_Timeout of 30 seconds.
